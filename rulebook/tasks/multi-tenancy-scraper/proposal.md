# Proposal: Multi-Tenancy Scraper Concurrency

## Why
The previous `MercadoLivreScraper` implementation used a strict Singleton pattern with a global `sessionLock`. This caused two major issues:
1. **Performance bottleneck**: Only one scraping operation could occur at a time across the entire Node.js process.
2. **Data leakage risk**: The Singleton shared browser cookies across all requests, meaning different users generating affiliate links could inadvertently use each other's affiliate cookies.

This change is required to support a true multi-tenant SaaS architecture where users execute concurrent collection jobs securely.

## What Changes
- Removed the strict Singleton pattern state from `MercadoLivreScraper` for page and cookies.
- Implemented `createIncognitoBrowserContext` using Puppeteer, ensuring that every collection or link generation request runs in an isolated, sandboxed environment with its own cookies and local storage.
- Replaced the global mutex `sessionLock` with a `launchLock` only for the browser process initialization, allowing unlimited parallel contexts inside the unified browser process.
- Updated `MercadoLivreService` with a `createForUser` factory method to fetch per-tenant API and cookie settings from the Database.
- Updated `CollectorService` endpoints to dynamically instantiate `MercadoLivreService` overriding default credentials.

## Impact
- Affected specs: `multi-tenancy`
- Affected code: `src/services/mercadolivre/MercadoLivreScraper.ts`, `src/services/mercadolivre/MercadoLivreService.ts`, `src/services/collector/CollectorService.ts`
- Breaking change: NO
- User benefit: Users can now execute parallel background scraping jobs. Zero risk of one user winning commissions on links generated by another user due to shared browser contexts.
